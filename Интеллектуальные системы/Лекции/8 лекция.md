
# 8 OLAP и технологии искусственного интеллекта

## 8.1 Концепция OLAP

### 8.1.1 Определение (Кодд, 1993)
> *«Технология оперативной аналитической обработки данных, использующая методы и средства для сбора, хранения и анализа многомерных данных в целях поддержки процессов принятия решений»*

### 8.1.2 Тест FASMI (1995) — требования к OLAP-системе

| Буква | Требование | Описание |
|-------|------------|----------|
| **F**ast | Быстрый отклик | < 5 секунд для большинства запросов |
| **A**nalysis | Аналитические вычисления | Числовые и статистические операции |
| **S**hared | Многопользовательский доступ | С блокировками и правами |
| **M**ultidimensional | Многомерное представление | Данные как куб |
| **I**nformation | Доступ ко всем нужным данным | Полный охват |

### 8.1.3 Ключевые правила Кодда (12 правил)

| № | Правило | Суть |
|---|---------|------|
| 1 | Многомерное концептуальное представление | Данные как гиперкуб |
| 2 | Прозрачность источника данных | Пользователь не видит, откуда данные |
| 3 | Доступность | OLAP поверх любой БД |
| 4 | Стабильная производительность | При росте данных не деградирует |
| 6 | Универсальность измерений | Все измерения равноправны |
| 12 | Неограниченное число измерений | ≥ 15, лучше 20 |

## 8.2 Операции над OLAP-кубом

| Операция | Описание | Результат | Применение |
|----------|----------|-----------|------------|
| **Slice** (срез) | Фиксация одного измерения | (N-1)-мерный срез | Анализ фиксированного периода |
| **Dice** (подкуб) | Диапазоны по нескольким измерениям | Меньший куб | Анализ сегмента |
| **Pivot** (поворот) | Перестановка осей куба | Та же структура, другой угол | Сводные таблицы |

## 8.3 Архитектуры OLAP: MOLAP, ROLAP, HOLAP

### 8.3.1 Сравнение архитектур

| Критерий | MOLAP | ROLAP | HOLAP |
|----------|-------|-------|-------|
| **Хранилище** | Куб (проприетарный) | Реляционная БД | Куб + Реляционная |
| **Скорость** | ★★★★★ | ★★☆☆☆ | ★★★★☆ |
| **Масштаб данных** | ★★☆☆☆ | ★★★★★ | ★★★★☆ |
| **Примеры** | Essbase, TM1 | Mondrian, MicroStrategy | SSAS, SAP BW |

### 8.3.2 MOLAP (Multidimensional OLAP)
- Данные в **многомерных массивах** (не SQL)
- **Максимальная скорость** запросов
- Предварительная агрегация при загрузке куба
- **Ограниченный объём** (сотни ГБ)
- **Проклятие размерности** — экспоненциальный рост объёма
- Чувствителен к изменениям модели

### 8.3.3 ROLAP (Relational OLAP)
- Куба **нет** — MDX-запросы транслируются в SQL
- Работает поверх реляционной схемы **«звезда»**
- **Масштабируется на терабайты**
- Медленнее при сложных агрегатах
- Современный выбор — **колончатые БД** (ClickHouse, BigQuery)

### 8.3.4 HOLAP (Hybrid OLAP)
- **Гибрид**: агрегаты верхних уровней (год, квартал) → MOLAP
- Детальные записи → ROLAP
- Динамическая маршрутизация запросов в зависимости от уровня детализации
- Оптимизирован под типичный паттерн: 90% запросов к агрегатам

## 8.4 Физические схемы данных для OLAP

### 8.4.1 Star Schema (Звезда)

```
┌─────────────┐
│  Dimension  │
│   (плоская) │
└──────┬──────┘
       │ FK
┌──────▼──────┐     ┌─────────────┐
│   FACT      │─────│  Dimension  │
│   (центр)   │     │   (плоская) │
└──────┬──────┘     └─────────────┘
       │ FK
┌──────▼──────┐
│  Dimension  │
│   (плоская) │
└─────────────┘
```

- Центр: **таблица фактов** (факты + внешние ключи)
- Лучи: **плоские таблицы измерений** (денормализованы)
- **Быстрее JOIN**, проще запросы, больше места
- Оптимально для большинства OLAP-задач

### 8.4.2 Snowflake Schema (Снежинка)
- Измерения **нормализованы до 3НФ**
- Меньше дублирования данных
- **Сложнее запросы** (больше JOIN-ов)
- Редко используется в современных ROLAP

### 8.4.3 В Lakehouse-архитектуре
- Таблицы фактов = Delta/Iceberg таблицы в **Gold-слое**
- Измерения = **SCD** (Slowly Changing Dimensions) с history tracking

## 8.5 Колончатое хранение и сжатие

### 8.5.1 Почему OLAP быстрее OLTP в 100×?

| OLTP (строчное хранение) | OLAP (колончатое хранение) |
|--------------------------|----------------------------|
| Читает всю строку целиком | Читает только нужные колонки |
| Много I/O при агрегациях | Минимум I/O |
| Сжатие ограничено | Высокое сжатие (однородные типы) |
| Оптимизирован под INSERT/UPDATE | Оптимизирован под SELECT/агрегаты |

## 8.6 Языки запросов

| Язык | Применение | Особенность |
|------|------------|-------------|
| **MDX** (Multidimensional Expressions) | MOLAP-кубы (SSAS, Essbase) | Нативный для многомерных данных |
| **SQL** | ROLAP-системы | Оконные функции = OLAP-операции |

### 8.6.1 Оконные функции SQL как OLAP в ROLAP
- `ROW_NUMBER()` — нумерация строк
- `LAG()/LEAD()` — доступ к предыдущим/следующим строкам
- `PERCENTILE_CONT()` — непрерывные процентили
- `SUM() OVER(PARTITION BY ...)` — агрегаты без GROUP BY

## 8.7 Иерархии измерений в OLAP

### 8.7.1 Определение
Способ организации данных измерения (dimension) в **уровни детализации**, позволяющий выполнять:
- **Drill-down** (углубление) — от общего к детальному
- **Roll-up** (свёртка) — от детального к общему

### 8.7.2 Примеры иерархий

| Измерение | Иерархия |
|-----------|----------|
| **Время** | Год → Квартал → Месяц → День |
| **География** | Страна → Регион → Город → Филиал |
| **Продукт** | Категория → Подкатегория → SKU |

### 8.7.3 Структура
- Каждое измерение может содержать **одну или несколько иерархий**
- Все измерения сходятся в **таблице фактов** (выручка, количество, и т.д.)

## 8.8 DuckDB: встраиваемый OLAP

### 8.8.1 Почему DuckDB важен для ML
- **In-process** — нет сервера, нет настройки (`import duckdb`)
- Нативно читает Parquet, CSV, JSON, Avro
- **Быстрее Pandas** для аналитических запросов в 10–100×
- Поддержка Python UDF + HuggingFace

### 8.8.2 Типичные сценарии
- Feature engineering на ноутбуке без кластера
- Exploration перед переходом на BigQuery/Snowflake
- CI/CD тестирование dbt-моделей локально
- **FAISS + DuckDB** — векторный поиск рядом с OLAP

### 8.8.3 Пример
```python
duckdb.sql("""
    SELECT cohort, AVG(ltv) OVER(PARTITION BY cohort) 
    FROM 'data/*.parquet'
""")
```

## 8.9 Откуда берутся признаки для ML? Роль OLAP

### 8.9.1 Проблема без OLAP
- Сырые транзакции → неудобный формат для ML
- Аналитик вручную считает агрегаты
- Разные команды → разный код → разные числа
- Нет переиспользования признаков

### 8.9.2 Решение: OLAP как Feature Source
- OLAP уже хранит все нужные агрегаты
- **Бизнес-метрики = готовые ML-признаки**
- Единый источник: BI и ML читают одно и то же
- Feature Store материализует агрегаты OLAP

> *Feature Engineering — это то, что аналитики делали вручную в OLAP десятилетиями. Feature Store автоматизирует этот процесс.*

---

## 8.10 Feature Store: роль OLAP

### 8.10.1 Архитектура

| Компонент | Назначение | Хранение | Пример |
|-----------|------------|----------|--------|
| **Offline Store** | Исторические признаки для обучения | Parquet в S3/GCS, point-in-time | Feast → FeatureView |
| **Online Store** | Низкая задержка для инференса (<10 мс) | Redis, DynamoDB | feast materialize-incremental |

### 8.10.2 Критическая проблема без Feature Store
> **Training-Serving Skew** — модель обучается на одних агрегатах, инференс получает другие.

**Решение**: OLAP как единый источник устраняет этот риск.

## 8.11 Семантический слой: единый словарь

### 8.11.1 Назначение
Одно определение метрики для **BI, OLAP и LLM-агентов**

### 8.11.2 Инструменты

| Инструмент | Особенность |
|------------|-------------|
| **dbt Semantic Layer** | Метрики в YAML, Git-версионирование, автоматические тесты |
| **Cube.dev (Cube.js)** | API для BI и приложений |

### 8.11.3 Пример dbt (metrics.yml)
```yaml
metrics:
  - name: revenue
    label: "Выручка"
    model: ref("orders")
    calculation_method: sum
    expression: "amount"
    filters:
      - field: status
        operator: "="
        value: "paid"
    dimensions:
      - name: region
      - name: product_category
      - name: order_date
        type: time
```

## 8.12 In-Database ML

### 8.12.1 Концепция
> *«Модель приходит к данным»* — устранение экспорта и задержки пайплайна

### 8.12.2 Сравнение подходов

| Традиционный подход (проблемы) | In-Database ML (решение) |
|-------------------------------|--------------------------|
| OLAP → экспорт CSV/Parquet → Python → модель | Обучение и инференс внутри хранилища |
| Утечка данных при экспорте (compliance риски) | Данные не покидают защищённый контур |
| Задержка пайплайна: часы на выгрузку терабайт | Нет задержки на экспорт |
| Несогласованность схем | Единая схема (те же таблицы, что для BI) |

> *Ключевая идея: данные слишком велики, чтобы их перемещать. Перемещаем вычисления.*

## 8.13 HTAP: конвергенция OLTP и OLAP

### 8.13.1 Определение
**Hybrid Transactional/Analytical Processing** — real-time inference без ETL-задержки

### 8.13.2 Проблема ETL-задержки
```
OLTP → ETL (часы) → OLAP → ML
```
- Рекомендация основана на **вчерашних** данных
- Признак обновится через N часов после транзакции
- Для fraud detection — **слишком поздно!**

### 8.13.3 HTAP-решения

| Решение | Особенность |
|---------|-------------|
| **TiDB** | TiKV (транзакции) + TiFlash (аналитика) |
| **SingleStore** | Row store + Column store, sub-second analytics |
| **SAP HANA** | In-memory HTAP |
| **Google AlloyDB** | PostgreSQL-совместимый HTAP с ML |
| **YugabyteDB** | Distributed HTAP с YSQL |

### 8.13.4 HTAP для ML
> Признак обновляется в момент транзакции → инференс на свежих данных → fraud detection в **12 мс** вместо T-hours

## 8.14 Проблема галлюцинаций LLM: почему нужен OLAP

### 8.14.1 Почему LLM галлюцинирует числа
- LLM обучен предсказывать следующий токен, **не хранить данные**
- Числа в тренировочных данных — **статистика, а не факты**
- Параметры модели **не обновляются** с новыми данными
- *Confidently wrong* — уверенный ответ ≠ правильный ответ

### 8.14.2 Архитектурное решение: OLAP-грундинг (grounding)

```
LLM генерирует SQL → OLAP выполняет → факт возвращается → LLM только форматирует
```

- LLM **не вычисляет агрегат**, только форматирует ответ
- SQL-запрос **прозрачен** — можно проверить и аудировать
- Семантический слой гарантирует правильное определение метрики
- Источник: **реальные транзакции**, не предположения модели

> *Никогда не доверяйте LLM числа, которые он не получил из верифицированного источника (OLAP, БД, API).*

## 8.15 Дерево выбора OLAP-инструмента

### 8.15.1 Облачный managed OLAP

| Инструмент | Когда выбирать | AI-интеграция |
|------------|----------------|---------------|
| **Snowflake** | Нужен Cortex AI / LLM в SQL | ★★★★★ |
| **BigQuery** | Google Cloud экосистема + Vertex AI | ★★★★★ |
| **Databricks SQL** | Уже используете Spark/Delta | ★★★★☆ |
| **Redshift** | AWS-стек, Redshift ML | ★★★☆☆ |

**Критерии**: нет DevOps, нужна AI-зрелость, бюджет > $10K/мес

### 8.15.2 Open Source / Self-hosted

| Инструмент | Когда выбирать | AI-интеграция |
|------------|----------------|---------------|
| **ClickHouse** | Высокая нагрузка, нужен контроль | ★★★☆☆ |
| **DuckDB** | Аналитика на ноутбуке, CI/CD тесты | ★★★★☆ |
| **Apache Druid** | Real-time OLAP + Kafka | ★★★☆☆ |
| **Apache Kylin** | Уже есть Hadoop-кластер | ★★★☆☆ |

**Критерии**: строгий бюджет или compliance, своя команда DevOps
