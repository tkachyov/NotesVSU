
# 7 Современные архитектуры данных и AI-специализированные хранилища

## 7.1 Эволюция архитектур (повторение и углубление)

| Архитектура | Период | Ключевые характеристики |
|-------------|--------|-------------------------|
| **DWH** | 1990-е | Структурированные данные, OLAP, ETL |
| **Data Vault** | 2000-е | Гибкость, историчность, Agile |
| **Data Lake** | 2010-е | Все типы данных, ELT, Schema-on-Read |
| **Data Lakehouse** | 2015+ | Lake + ACID + SQL = единая платформа |
| **Data Fabric** | 2018+ | Виртуализация + AI + метаданные |
| **Data Mesh** | 2020+ | Децентрализация, домены, self-serve |

---

## 7.2 Data Lake (озеро данных)

### 7.2.1 Определение (Дж. Диксон, 2010)
- **Централизованное хранилище** для данных **ЛЮБОГО типа** в **ИСХОДНОМ формате**
- **Schema-on-Read**: схема интерпретируется при чтении (а не при записи)

### 7.2.2 Ключевые особенности
- Хранение неструктурированных, полуструктурированных и структурированных данных
- Низкая стоимость хранения
- Поддержка Big Data и ML-нагрузок

---

## 7.3 Data Lakehouse

### 7.3.1 Концепция
- Объединяет преимущества **Data Lake** (все типы данных, низкая стоимость) и **DWH** (ACID, SQL)
- **Lake + ACID + SQL = единая платформа**

### 7.3.2 Ключевые технологии
- **Форматы таблиц**: Delta Lake, Apache Iceberg, Apache Hudi
- Обеспечивают ACID-транзакции поверх объектного хранилища

---

## 7.4 Медальонная архитектура (Medallion)

Структура обработки данных в Data Lake / Lakehouse — разделение на три последовательных слоя:

| Слой | Назначение |
|------|------------|
| **Bronze** (бронза) | Сырые данные как есть (источник истины) |
| **Silver** (серебро) | Очищенные, валидированные, нормализованные данные |
| **Gold** (золото) | Агрегированные, обогащённые данные для бизнес-аналитики и ML |

---

## 7.5 Data Fabric (фабрика данных)

### 7.5.1 Определение
Комбинация архитектуры, технологий и сервисов для управления множеством типов данных с использованием нескольких систем на разных платформах.

### 7.5.2 Ключевые компоненты
- **Виртуализация данных** — единая точка доступа, API для интеграции источников
- **MDM** (Master Data Management)
- **Knowledge Graph** — семантические сети
- **AI/ML** для оптимизации управления данными
- **Data Governance** — политики и аудит

### 7.5.3 Цель
> Единое информационное пространство поверх разнородных источников **без физического копирования данных**

### 7.5.4 Применение
- Реализуется через готовые платформы (Arenadata EDP и др.)
- Упрощает доступ к данным независимо от местоположения и формата

---

## 7.6 Data Mesh

### 7.6.1 Концепция
- **Децентрализация управления данными** (в отличие от DWH и Data Lake)
- Распределяет владение данными по **доменам**

### 7.6.2 Четыре принципа Data Mesh

| Принцип | Описание |
|---------|----------|
| **Доменное владение** | Продажи, финансы, производство — каждая команда владеет своими данными |
| **Данные как продукт** | Каждый домен производит «дата-продукт» с гарантиями качества |
| **Self-serve платформа** | Инфраструктура для самостоятельной работы с данными |
| **Федеративное управление** | Единые стандарты качества и безопасности при децентрализации |

### 7.6.3 Преимущества и ограничения

| Преимущества | Ограничения |
|--------------|-------------|
| Распределённое управление | Сложность интеграции и согласованности |
| Гибкий доступ к данным | Только для крупных корпораций |
| Масштабирование команд и инфраструктуры | Требует изменения оргструктуры и бизнес-процессов |

---

## 7.7 Сравнение современных архитектур

| Критерий | Data Warehouse | Data Lake | Data Lakehouse | Data Mesh |
|----------|---------------|-----------|----------------|-----------|
| **Типы данных** | Структурированные | Все типы | Все типы | Все типы |
| **Гибкость** | Управляемая | Высокая | Высокая | Очень высокая |
| **Управление** | Централизованное | Централизованное | Централизованное | Децентрализованное |
| **Обработка** | ETL | ELT (сырые) | Гибкое ELT | Доменно-ориентированная |
| **Применение** | BI, отчётность | ML, Big Data | ML + BI + Streaming | Крупные организации |
| **ACID** | Да | Нет | Да (Delta/Iceberg) | Зависит от платформы |

# 8 AI-специализированные хранилища

## 8.1 Object Storage (объектное хранилище)

### 8.1.1 Архитектурные принципы
- Объект = содержимое + метаданные + уникальный ID
- **Плоское адресное пространство** (нет иерархии папок)
- Неограниченная масштабируемость
- **11×9 durability** (S3: 99.999999999%)
- **Tiered storage**: Standard → IA (Infrequent Access) → Glacier

### 8.1.2 Представители
- **Amazon S3** — стандарт де-факто
- **Google Cloud Storage** — интеграция с Vertex AI
- **Azure Blob Storage** — ADLS Gen2
- **MinIO** — S3-совместимый, open source
- **DVC** (Data Version Control) — версионирование датасетов

### 8.1.3 Типичная структура ML-проекта в Object Storage
```
raw-data/          → изображения, аудио, видео, текст
processed-data/    → train / val / test
features/          → предрассчитанные признаки
models/            → .h5, .pt, SavedModel, ONNX
experiments/       → артефакты MLflow
annotations/       → разметка
```

## 8.2 Time-Series Databases (TSDB) — базы данных временных рядов

### 8.2.1 Характеристики нагрузки
- Преобладание записи над чтением
- Последовательная запись
- Агрегации по временным интервалам
- Неизменяемость после записи

### 8.2.2 Представители
| TSDB | Особенность |
|------|-------------|
| InfluxDB | Flux, высокая скорость записи |
| TimescaleDB | PostgreSQL + SQL |
| Apache Druid | OLAP real-time |
| Prometheus | Мониторинг |
| VictoriaMetrics | Высокое сжатие |

### 8.2.3 Применение в ИИ
- Прогнозирование спроса и нагрузки
- **Предиктивное обслуживание** (IoT)
- Обнаружение аномалий
- **Мониторинг ML-моделей** (data drift, concept drift)

### 8.2.4 Мониторинг ML-моделей с TSDB
```
Data drift → TSDB фиксирует метрики в реальном времени 
→ алерты при деградации модели 
→ автоматическое переобучение или эскалация
```

## 8.3 Vector Databases (векторные базы данных)

### 8.3.1 Эмбеддинги (Embeddings)
- Представление объекта в виде вектора фиксированной размерности
- **Ключевое свойство**: семантическая близость ≈ геометрическая близость
- Пример: "Кот сидит на коврике" → [0.12, -0.45, 0.78, ..., 0.33]

### 8.3.2 Размерности эмбеддингов
| Модель | Размерность |
|--------|-------------|
| BERT | 768 |
| CLIP | 512 |
| OpenAI Ada-002 | 1 536 |
| OpenAI emb-3-large | 3 072 |

### 8.3.3 Пайплайн векторного поиска
```
Объект → Энкодер → Вектор → Vector DB → Поиск ближайших соседей (ANN)
```

### 8.3.4 Алгоритмы приближённого поиска (ANN)
- **IVF** (Inverted File Index)
- **HNSW** (Hierarchical Navigable Small World)
- **PQ** (Product Quantization)
- **ScaNN** (Scalable Nearest Neighbors)

## 8.4 Feature Store

### 8.4.1 Решаемые проблемы
| Проблема | Описание |
|----------|----------|
| **Training-Serving Skew** | Разные стеки обработки признаков при обучении и инференсе → деградация модели |
| **Дублирование признаков** | Одна и та же логика признаков реализуется в разных командах |
| **Data Leakage** | Нарушение point-in-time correctness (использование будущих данных) |

### 8.4.2 Архитектура Feature Store

| Компонент | Назначение |
|-----------|------------|
| **Feature Registry** | Каталог признаков, метаданные, lineage, документация |
| **Offline Store** | Исторические значения (Data Lake/DWH), point-in-time join, обучающие выборки |
| **Online Store** | Последние значения (Redis/DynamoDB), латентность < 10 мс, real-time inference |
| **Materialization** | Вычисление признаков: Batch (Spark), Streaming (Flink), синхронизация offline→online |

### 8.4.3 Реализации
- **Open Source**: Feast
- **Managed**: Tecton, Databricks FS, SageMaker FS, Vertex AI FS, Hopsworks

## 8.5 Потоковые системы: Kafka и Flink

### 8.5.1 Apache Kafka
- Распределённая потоковая платформа
- **Топики → партиции → продюсеры/консьюмеры**
- **Роль в ИИ**: транспорт событий для Feature Engineering, буферизация пиков, Event Sourcing

### 8.5.2 Apache Flink
- Потоковая обработка с **exactly-once семантикой**
- **Event time + watermarks** для корректной агрегации
- **Роль в ИИ**: real-time feature engineering, скользящие агрегаты → Online Feature Store

### 8.5.3 Real-time ML Pipeline
```
События → Kafka → Flink → Feature Store (Online) → Model Serving → Предсказание
```

## 8.6 Управление моделями: MLflow

### 8.6.1 MLflow Tracking
| Тип | Примеры |
|-----|---------|
| **Parameters** | Гиперпараметры |
| **Metrics** | Accuracy, F1, loss |
| **Artifacts** | Модели, графики |
| **Meta** | Время, автор, теги |
| **Backend Store** | PostgreSQL / MySQL |
| **Artifact Store** | S3 / GCS / ADLS |

### 8.6.2 MLflow Model Registry
- Каталог версий моделей
- **Стадии**: None → Staging → Production → Archived
- Полная трассируемость: для каждой Production-модели известны эксперимент, данные, гиперпараметры, метрики

### 8.6.3 Альтернативы
| Инструмент | Специализация |
|------------|---------------|
| Weights & Biases | Визуализация |
| Neptune.ai | Командная работа |
| ClearML | End-to-end MLOps |
| DVC | Версионирование данных + моделей |

## 8.7 Хранилища для LLM-приложений

| Тип хранилища | Назначение | Примеры |
|---------------|------------|---------|
| **Embedding Store** | Vector DB для RAG (документы → чанки → эмбеддинги → поиск) | Pinecone, Weaviate, Qdrant, Milvus |
| **Conversation Memory** | История диалогов по session_id | Redis, DynamoDB |
| **Semantic Cache** | Кэш ответов LLM на близкие запросы | Vector DB |
| **Knowledge Graph** | Факты для GraphRAG, повышение точности | Neo4j, Neptune |

### 8.7.1 Интеграционные фреймворки
**LangChain / LlamaIndex** — позволяют заменить реализацию хранилища (VectorStore, DocumentStore, Memory, Cache) без изменения бизнес-логики.

## 8.9 Критерии выбора хранилища

### 8.9.1 По типу данных
| Тип данных | Рекомендуемое хранилище |
|------------|------------------------|
| Табличные | DWH / Lakehouse |
| Изображения, аудио | Object Storage |
| Текст для LLM | Vector DB |
| Графы | Graph DB |
| Временные ряды | TSDB |

### 8.9.2 По латентности
| Требование | Решение |
|------------|---------|
| Batch (минуты) | Data Lake, DWH |
| Near real-time | Streaming + Feature Store |
| Real-time (мс) | Redis, Vector DB |

### 8.9.3 По масштабу
| Объём данных | Решение |
|--------------|---------|
| < 100 GB | PostgreSQL, ChromaDB |
| 100 GB – 10 TB | Managed DWH |
| > 10 TB | Lakehouse, распределённые |

### 8.9.4 По стадии проекта
| Стадия | Решение |
|--------|---------|
| Эксперимент | SQLite, ChromaDB (локально) |
| MVP | Managed сервисы |
| Production | Полная платформа |
