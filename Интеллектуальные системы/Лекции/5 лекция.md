
# 5 Загрузка, обогащение данных и хранилища для ИИ

## 5.1 Загрузка данных в хранилище (ETL)

### 5.1.1 Место загрузки в ETL-процессе
- Данные извлечены из источников
- Выполнены преобразование, агрегация и очистка
- **Последний этап** — перенос данных из промежуточных таблиц в структуры хранилища данных (ХД)

### 5.1.2 Типы источников данных при загрузке
| Обозначение | Расшифровка |
|-------------|-------------|
| **SCF** | Структурированный текстовый файл |
| **UCF** | Пользовательский классификатор |
| **MLR** | RDS или таблица фактов оперативной системы |
| **DWH** | Другое хранилище данных |

---

## 5.2 Неполная загрузка данных

### 5.2.1 Причины отказа в загрузке записей
- Не удалось исправить критические ошибки на этапе преобразования
- Некорректный порядок загрузки (факты для измерения, которое ещё не загружено)
- Внутренние проблемы ХД (недостаток места)
- Прерывание процесса или остановка пользователем

### 5.2.2 Результат
- Некоторые записи не попадают в хранилище
- Предусмотрена система **извещения пользователя** о неполноте загруженных данных

---

## 5.3 Streaming ETL (потоковый ETL)

### 5.3.1 Определение
Обработка и перемещение данных в **реальном времени** из одного места в другое.

### 5.3.2 Применение
- Обнаружение мошенничества
- Интернет вещей (IoT)
- Потоковая аналитика
- Обработка платежей в реальном времени

### 5.3.3 Особенность
Приложения реального времени требуют **немедленной реакции** на новые данные по мере их создания.

---

## 5.4 Обогащение данных (Data Enrichment)

### 5.4.1 Проблематика
- Данные для анализа должны быть **полными**, **достоверными** и **информационно насыщенными**
- **Важное различие**:
  - **Данные** — объективны (существуют как изменения физического процесса)
  - **Информация** — субъективна (зависит от компетенции эксперта)

### 5.4.2 Определение обогащения
Процесс насыщения данных новой информацией, делающей их более **ценными и значимыми** для аналитической задачи.

### 5.4.3 Два основных метода

| Метод | Описание |
|-------|----------|
| **Внешнее обогащение** | Привлечение информации из внешних источников |
| **Внутреннее обогащение** | Использование собственных данных предприятия |

### 5.4.4 Источники для внешнего обогащения
- Другие предприятия (партнёры, конкуренты)
- Финансово-кредитные учреждения, банки, страховые компании
- Государственные налоговые и статистические службы
- Органы государственного и муниципального управления
- Службы социальной сферы (миграция, труд, здравоохранение, пенсионные фонды)

### 5.4.5 Пример задачи из лекции
**Постановка**: выработать политику взаимодействия с поставщиками в зависимости от их надёжности.

**Решение**:
1. Разработать критерии надёжности
2. Рассчитать метрику для каждого поставщика (дни задержки, стоимость поставки)
3. Разбить поставщиков на 3 категории
4. Добавить поле в таблицу хранилища
5. Использовать в дальнейшем анализе

## 5.5 ELT (Extract, Load, Transform)

### 5.5.1 Отличие от ETL
| Характеристика | ETL | ELT |
|----------------|-----|-----|
| Порядок | Extract → Transform → Load | Extract → Load → Transform |
| Место преобразования | В конвейере до загрузки | В хранилище после загрузки |
| Гибкость | Ниже | Выше |
| Подход к данным | Очистка до загрузки | Загружаются любые данные |

### 5.5.2 Когда подходит ELT
- **Огромные объёмы данных** (структурированных и неструктурированных)
- **Наличие вычислительной мощности** (облачные решения)
- **Приоритет скорости передачи** — все данные (хорошие, плохие) попадают в озеро данных для последующего преобразования

### 5.5.3 Инструмент
**Apache Airflow** — платформа с открытым кодом для создания и управления конвейерами обработки данных, включая ELT.

## 5.6 Streaming AI (real-time inference)
- В real-time сценариях используется:
  - **Streaming ETL** (Kafka, Flink)
  - **Online feature computation**
  - **Materialized views** для low-latency inference

## 5.7 FEL (Feature Engineering Layer)

### 5.7.1 Входные и выходные данные FEL
- **Вход**: очищенные, согласованные таблицы фактов и измерений из ODS или Staging Area (свободны от технических артефактов)
- **Выход**: Feature Table или Feature Store — структурированный, версионированный, документированный набор признаков (описание, тип, статистики, история изменений)

### 5.7.2 Основные процессы FEL

| Процесс | Описание |
|---------|----------|
| **Feature Selection** (отбор признаков) | Выбор наиболее информативных признаков → устранение шума, снижение размерности, уменьшение переобучения |
| **Feature Transformation** (трансформация) | Приведение данных к требуемым свойствам (симметричное распределение, отсутствие выбросов, единые масштабы) |
| **Feature Construction** (конструирование) | Синтез новых признаков из существующих (наиболее творческий, доменно-зависимый процесс) |

---

## 5.8 Feature Store (Хранилище признаков)

### 5.8.1 Ключевые возможности

| Возможность | Описание |
|-------------|----------|
| **Версионирование признаков** | Каждый признак имеет версию, описание, схему, историю. Эксперимент привязан к конкретной версии. |
| **Feature Registry (каталог)** | Централизованный реестр всех признаков с документацией, владельцами, статистиками |
| **Lineage (происхождение)** | Фиксация источников, трансформаций, авторов изменений |
| **Offline/Online consistency** | Гарантия идентичности признака при обучении и инференсе (устранение training-serving skew) |
| **Мониторинг дрейфа** | Автоматическое обнаружение data drift и concept drift |
| **Scheduled recomputation** | Автоматический пересчёт признаков по расписанию (Airflow, Prefect) |

## 5.9 Хранилища данных в системах ИИ

### 5.9.1 Место хранилищ в AI-архитектуре (снизу вверх)

```
AI/ML ПРИЛОЖЕНИЯ (чат-боты, рекомендации, CV, NLP)
              ↑
МОДЕЛИ И MLOps (LLMs, Feature Stores, мониторинг)
              ↑
ELT / DATA PIPELINES (трансформация, оркестрация, качество данных)
              ↑
ХРАНЕНИЕ ДАННЫХ (Lakehouse, Vector DB, DWH, Object Storage) ← ТЕМА ЛЕКЦИИ
              ↑
ETL / DATA PIPELINES (трансформация, оркестрация, качество данных)
              ↑
ИСТОЧНИКИ ДАННЫХ (Databases, APIs, Streams, Files, IoT)
```

### 5.9.2 Данные как фундамент систем ИИ

**Цитата (Sculley et al., Google, NeurIPS 2015)**:
> *"Собственно код модели составляет не более 5% от всей кодовой базы промышленной ML-системы"*

**Остальные 95% включают**:
- Сбор и валидация данных
- Хранение и трансформация
- Конструирование признаков
- Управление конфигурациями
- Мониторинг и обслуживание

### 5.9.3 Ключевой вывод
> Выбор хранилища определяет **производительность**, **воспроизводимость** и саму **возможность промышленной эксплуатации** системы ИИ.

### 5.9.4 Эволюция требований
Каждое поколение AI-задач предъявляло новые требования — от структурированных таблиц до **семантического поиска по эмбеддингам**.

## 5.10 Итоговые определения для запоминания

| Термин | Суть |
|--------|------|
| **ETL** | Transform до загрузки, классический подход |
| **ELT** | Transform после загрузки, для больших объёмов и неструкт. данных |
| **Streaming ETL** | Реальное время, Kafka/Flink |
| **Data Enrichment** | Насыщение данных новой информацией (внешнее/внутреннее) |
| **FEL** | Feature Engineering Layer (отбор, трансформация, конструирование) |
| **Feature Store** | Централизованное, версионированное хранилище признаков |
| **Training-serving skew** | Рассогласование признаков при обучении и инференсе (устраняется Feature Store) |
