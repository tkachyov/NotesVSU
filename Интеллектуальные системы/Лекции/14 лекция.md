# 14 Продукционные системы и управление выводом

## 14.1 Мотивация: почему правила?

### 14.1.1 Проблема прямого программирования

**Сценарий**: система диагностики инфекционного заболевания по симптомам пациентов.

| Прямое программирование | Продукционные правила |
|------------------------|----------------------|
| Тысячи вложенных условий (`IF fever AND cough AND ...`) | Каждое правило — независимый модуль знания |
| Добавить симптом = переписать код | Добавить правило = добавить строку в базу |
| Невозможно объяснить ход рассуждения | Трассировка вывода — встроенная функция |
| Знания эксперта = алгоритм → хрупкость (brittle) | Система отделяет знания от механизма рассуждения |

> **Ключевая идея**: продукционная система отделяет **знания** от **механизма рассуждения**

## 14.2 Продукция: IF условие THEN действие

### 14.2.1 Синтаксис правила (CLIPS-стиль)

```
(defrule имя-правила
    (условие-1)
    (условие-2)
    ...
    =>
    (действие-1)
    (действие-2)
    ...)
```

### 14.2.2 Примеры правил

**Правило R1 — диагноз**:
```clips
(defrule diagnose-flu
    (температура ?t && (> ?t 38.5))
    (симптом кашель)
    (симптом слабость)
    =>
    (assert (диагноз грипп)))
```

**Правило R2 — рекомендация**:
```clips
(defrule recommend
    (диагноз грипп)
    =>
    (assert (лечение парацетамол))
    (printout t "Рекомендовано: парацетамол"))
```

### 14.2.3 Элементы левой части (LHS)

| Элемент | Описание | Пример |
|---------|----------|--------|
| **Шаблон факта** | Сопоставление с WM | `(температура ?t)`, `(возраст ?a)` |
| **Тест** | Условие на переменные | `(> ?t 38.5)`, `(> ?a 12)` |
| **Отрицание** | Факт отсутствует | `(not (диагноз грипп))` |

## 14.3 Формальная модель продукционной системы

**Продукционная система = (WM, RB, IE)**

| Компонент | Название | Назначение |
|-----------|----------|------------|
| **WM** | Working Memory (рабочая память) | Хранит текущие факты (симптомы, диагнозы, рекомендации) |
| **RB** | Rule Base (база правил) | Хранит продукционные правила |
| **IE** | Inference Engine (механизм вывода) | Управляет циклом «сопоставление → выбор → выполнение» |

### 14.3.1 Цикл работы Inference Engine

```
1. **Match** (сопоставление) — найти все правила, чьи условия (LHS) соответствуют текущим фактам WM
2. **Conflict resolution** (разрешение конфликтов) — выбрать одно правило из конфликтного набора
3. **Act** (выполнение) — выполнить действия правила (THEN), изменить WM
4. Перейти к шагу 1 (пока не достигнуто условие остановки)
```

## 14.4 Применение прямого и обратного вывода

| Предметная область | Прямой вывод (Forward Chaining) | Обратный вывод (Backward Chaining) |
|-------------------|--------------------------------|-----------------------------------|
| **Медицинская диагностика** | MYCIN: от симптомов к диагнозу | Проверка конкретной гипотезы врача |
| **Конфигурирование систем** | XCON/R1: применять правила к спецификации | Проверить выполнимость требования |
| **Мониторинг и алерты** | Drools: события → правила → действия | «Почему сработал алерт?» (объяснение) |
| **Юридический анализ** | Редко (слишком много фактов) | «Применима ли статья 159 УК?» — цель известна |

---

## 14.5 Пространство поиска

- Каждое применение правила — это **переход** между состояниями WM
- Пространство состояний = все возможные наборы фактов в WM
- Задача вывода = найти путь от начального WM к целевому

---

## 14.6 Конфликтный набор (Conflict Set)

**Проблема**: на каждом шаге цикла может «сработать» сразу несколько правил.

**Конфликтный набор** — множество всех правил, чьи условия (LHS) удовлетворены текущим состоянием WM.

**Решение**: стратегии разрешения конфликтов (выбор одного правила).

---

## 14.7 Стратегии разрешения конфликтов

### 14.7.1 Recency (новизна фактов)

**Принцип**: приоритет правилам, использующим наиболее **свежие** факты в WM.

| Достоинства | Недостатки |
|-------------|------------|
| Интуитивна («свежие данные важнее») | Может игнорировать важные старые факты |
| Хороша для event-driven систем | Опасна при накопленных противоречиях |
| По умолчанию в большинстве ЭС | Не учитывает семантику правил |

### 14.7.2 Specificity (специфичность)

**Принцип**: приоритет более **специфичному** правилу (с бо́льшим числом условий в LHS).

**Пример**:

| Правило | Условия | Действие |
|---------|---------|----------|
| `R_общее` | 1 условие (`симптомы ≥ 1`) | `assert (болен)` |
| `R_специфичное` | 3 условия (температура > 38.5 + кашель + ПЦР положит.) | `assert (грипп А)` |

→ **Specificity выбирает R_специфичное** (3 > 1 условие)

| Достоинства | Недостатки |
|-------------|------------|
| Отражает логику эксперта | Сложнее вычислять при большой БЗ |
| Естественна для медицины и права | Может конфликтовать с Recency |
| Уменьшает ложные срабатывания | Требует аккуратного проектирования LHS |

### 14.7.3 Salience (явный приоритет)

**Принцип**: явно заданный числовой приоритет (чем выше, тем важнее).

- **Детерминирована**: да
- **Применение**: бизнес-правила, критические системы

### 14.7.4 MEA (Means-Ends Analysis)

**Принцип**: сокращение разрыва между текущим состоянием и целевым.

- **Детерминирована**: да
- **Применение**: планирование, GPS-подобные системы

### 14.7.5 LEX

**Принцип**: Salience → Recency → Specificity (комбинация)

- **Детерминирована**: да
- **Применение**: OPS5, классические ЭС

### 14.7.6 Random

**Принцип**: случайный выбор из конфликтного набора

- **Детерминирована**: нет
- **Применение**: тестирование, отладка БЗ

---

## 14.8 Сравнение стратегий разрешения конфликтов

| Стратегия | Критерий выбора | Детерминирована? | Применение |
|-----------|----------------|------------------|------------|
| **Recency** | Свежесть фактов в WM | Нет | Мониторинг, event-driven |
| **Specificity** | Число условий в LHS | Да | Медицина, право, диагностика |
| **Salience** | Явный числовой приоритет | Да | Бизнес-правила, критические системы |
| **MEA** | Сокращение разрыва цель-состояние | Да | Планирование, GPS |
| **LEX** | Salience → Recency → Specificity | Да | OPS5, классические ЭС |
| **Random** | Случайный выбор | Нет | Тестирование, отладка |

### 14.8.1 Практические реализации

| Система | Стратегия по умолчанию |
|---------|------------------------|
| **CLIPS** | Salience + Recency |
| **Drools** | Salience + Complexity |
| **Prolog** | DFS (first-match) |

## 14.9 «Зима ИИ»: причины и уроки (1987–1993)

Обвал рынка экспертных систем произошёл по следующим причинам:

| Причина | Описание |
|---------|----------|
| **Хрупкость знаний** | ЭС работали только в узко определённой области. Малейший сдвиг → пересмотр всей базы правил. |
| **Стоимость разработки** | Требовались дорогостоящие инженеры знаний и месяцы работы с экспертами. |
| **Аппаратная зависимость** | Лидирующие ЭС работали на Lisp Machines ($100K+). Когда PC стали мощнее — они стали не нужны. |
| **Завышенные ожидания** | Гипербола 1984–86: «ЭС заменят врачей, юристов, инженеров». |
| **Отсутствие обучения** | ЭС не улучшались с опытом. Нейросети 1990-х начали демонстрировать обучение. |
| **Проблема поддержки** | «Знание устаревает». Компания менялась, а ЭС — нет. |

### 14.9.1 Урок

> **ЭС не умерли** — они превратились в **BRMS** (Business Rules Management Systems) и **XAI-компоненты**

## 14.10 Продукционные системы vs нейросети

| Критерий | Продукционные системы | Нейросети |
|----------|----------------------|-----------|
| **Знание** | Явное: правила IF-THEN | Неявное: веса нейронов |
| **Объяснимость (XAI)** | Полная трассировка вывода | Black box (без XAI-методов) |
| **Обучение** | Ручное (инженер знаний) | Автоматическое (данные) |
| **Работа с малыми данными** | Отлично (данные не нужны) | Плохо (без больших датасетов) |
| **Гибкость / обобщение** | Ограничена областью правил | Высокая при достатке данных |
| **Детерминированность** | Да (одинаковый вход → вывод) | Нет (стохастичность) |
| **Изменение знаний** | Добавить / удалить правило | Переобучить модель |
| **Регуляторные требования** | Полностью аудируемы | Сложно без XAI-надстроек |

## 14.11 Продукционные системы сегодня

### 14.11.1 XAI — объяснимый ИИ

- **EU AI Act (2024)**: высокорисковые системы должны быть объяснимы
- Продукционные правила — **native XAI**: трассировка = объяснение
- **Гибрид**: нейросеть предсказывает + правила объясняют
- **Применение**: кредитный скоринг, медицина, найм

### 14.11.2 BPM — бизнес-процессы

- **Drools Business Central**: визуальный редактор правил для аналитиков
- **DMN (Decision Model Notation)** — стандарт OMG для бизнес-решений
- **Применение**: банки, страхование, телеком (миллионы правил в продакшене)
- **Ключевое**: правила пишут аналитики, а не программисты

### 14.11.3 ИИ-агенты и планировщики

- Продукционные правила задают **политику агента**
- **PDDL** (Planning Domain Definition Language) — наследник OPS5
- **Роботизированные системы**: правила безопасности + нейросеть
- **Игровой ИИ**: Behavior Trees + правила для детерминированных решений
