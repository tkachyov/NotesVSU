
# 6 Хранилища данных: архитектуры, СУБД и эволюция подходов

## 6.1 Предпосылки создания хранилищ данных (DWH)

### 6.1.1 Исторический контекст
- Организации накапливают большие объёмы электронных данных
- Данные хранят потенциал для извлечения аналитической информации
- Потребность в анализе осознана руководством в **1980–90-х гг.**

### 6.1.2 Фундаментальное противоречие

| Характеристика | Оперативные системы (OLTP) | Системы анализа (OLAP/DWH) |
|----------------|----------------------------|----------------------------|
| **Цель** | Производительность транзакций | Гибкость и широкие возможности |
| **Данные** | Текущие | Исторические (годы) |
| **Обновления** | Маленькие порции | Большие порции загрузки |
| **Объём** | Сотни МБ — ГБ | ГБ — ТБ и выше |

---

## 6.2 Реляционные СУБД (РСУБД)

### 6.2.1 Архитектурные основы (Э. Кодд, 1970)
- Данные хранятся в **таблицах** (отношениях)
- **Нормализация** схемы
- **Ограничения целостности**
- **ACID**: Атомарность, Согласованность, Изолированность, Долговечность
- **Представители**: PostgreSQL, MySQL, Oracle, MS SQL Server

### 6.2.2 Роль в системах ИИ
- **Источник данных** для обучения (CRM, ERP → SQL → выборка)
- **Хранилище метаданных** ML-инфраструктуры (MLflow, Airflow → PostgreSQL)
- **In-Database ML** (Oracle ML, SQL Server ML Services, MADlib)
- **Векторный поиск**: gvector — тип VECTOR, индексы IVFFlat / HNSW

### 6.2.3 Ограничения
- Жёсткая схема
- Вертикальное масштабирование
- Не оптимально для неструктурированных данных

---

## 6.3 Нереляционные СУБД (NoSQL)

### 6.3.1 Теорема CAP (Э. Брюер)
Распределённая система не может одновременно обеспечить:
- **C**onsistency (Согласованность)
- **A**vailability (Доступность)
- **P**artition tolerance (Устойчивость к разделению)

> **NoSQL**: приоритет A+P, допущение **Eventual Consistency** (итоговой согласованности)

### 6.3.2 Типы NoSQL и их применение в ИИ

| Тип | Пример | Применение в ИИ |
|-----|--------|-----------------|
| **Document** | MongoDB | Каталоги датасетов, конфигурации моделей, журналы экспериментов (гибкая схема JSON) |
| **Key-Value** | Redis | Кэш предсказаний, Online Feature Store (<10 мс), очереди задач (суб-мс латентность) |
| **Column-Family** | Cassandra | Поведенческие данные, IoT, результаты инференса (LSM-tree, высокая скорость записи) |
| **Graph DB** | Neo4j | Knowledge Graphs (GraphRAG), GNN, рекомендации, обнаружение мошенничества (узлы + рёбра) |

## 6.4 ETL-процесс: основа наполнения DWH
- При разработке DWH **не менее 60%** всех затрат связано с переносом данных
- ETL — ключевой компонент (Extract, Transform, Load)

## 6.5 Метаданные в хранилищах данных
- **Метаданные** («данные о данных») — совокупность спецификаций и данных, описывающих:
  - Содержание
  - Структуру
  - Правила обработки данных в DWH

## 6.6 Data Vault: методология моделирования DWH

### 6.6.1 Определение
**Data Vault** — детально ориентированный, исторически отслеживаемый набор нормализованных таблиц для крупномасштабных хранилищ данных.

### 6.6.2 Три ключевых элемента

| Элемент | Описание |
|---------|----------|
| **Hub** | Бизнес-ключи (уникальные идентификаторы сущностей) |
| **Link** | Связи между Hub (ассоциации) |
| **Satellite** | Атрибуты и исторические изменения Hub или Link |

### 6.6.3 Загрузка данных в 2 шага (vs многоступенчатый JOIN в классических БД)
1. Загрузка всех **Hub** параллельно
2. Загрузка всех **Link** и **Satellite** (ссылаются только на уже загруженные Hub)

### 6.6.4 Преимущества и ограничения

| Преимущества | Ограничения |
|--------------|-------------|
| Гибкость: новые сущности = новый Hub, без изменения существующих | Большое количество JOIN-ов при запросах |
| Историчность: изменения фиксируются в Satellite | Обязательное наличие витрин данных |
| Параллельная разработка несколькими командами | Избыточное количество таблиц |
| Agile «из коробки»: поэтапное добавление элементов | |
| Отказоустойчивость: недоступный Satellite не блокирует систему | |
| Быстрая загрузка: всего 2 шага | |

### 6.6.5 Когда применять Data Vault
- Много разнородных источников (микросервисы)
- Активный Agile, частые изменения требований
- Требуется параллельная разработка
- Быстрые JOIN-ы (Snowflake, Greenplum, Vertica)

## 6.7 Эволюция архитектур: от DWH к Data Mesh

### 6.7.1 Хронология архитектур

| Архитектура | Период | Ключевые характеристики |
|-------------|--------|-------------------------|
| **DWH** | 1990-е | Структурированные данные, OLAP, ETL |
| **Data Vault** | 2000-е | Гибкость, историчность, Agile |
| **Data Lake** | 2010-е | Все типы данных, ELT, Schema-on-Read |
| **Data Lakehouse** | 2015+ | Lake + ACID + SQL = единая платформа |
| **Data Fabric** | 2018+ | Виртуализация + AI + метаданные |
| **Data Mesh** | 2020+ | Децентрализация, домены, self-serve |

### 6.7.2 Драйверы эволюции
- Рост объёма данных
- Появление **неструктурированных данных** (2000–2010-е)
- Потребность в новых архитектурных решениях

## 6.8 Итоговая таблица сравнения СУБД для ИИ-систем

| Тип | Примеры | Сильные стороны | Слабые стороны | Применение в ИИ |
|-----|---------|----------------|----------------|-----------------|
| **Реляционные (SQL)** | PostgreSQL, MySQL, Oracle | ACID, стандартизация, зрелость | Жёсткая схема, вертик. масштабирование | Источники данных, метаданные MLflow |
| **Document** | MongoDB | Гибкая схема JSON | Нет сложных JOIN | Каталоги датасетов, конфигурации |
| **Key-Value** | Redis | Суб-мс латентность | Ограниченные запросы | Online Feature Store, кэш |
| **Column-Family** | Cassandra | Высокая скорость записи | Сложные агрегации | IoT, результаты инференса |
| **Graph** | Neo4j | Нативные связи | Распределение | Knowledge Graphs, GNN |

## 6.9 Ключевые термины лекции

| Термин | Определение |
|--------|-------------|
| **OLTP** | Online Transaction Processing — оперативная обработка транзакций |
| **OLAP** | Online Analytical Processing — аналитическая обработка |
| **ACID** | Atomicity, Consistency, Isolation, Durability |
| **CAP теорема** | Распределённые системы: C, A, P — выбрать два |
| **Eventual Consistency** | Итоговая согласованность (компромисс NoSQL) |
| **Data Vault** | Методология моделирования для крупных DWH |
| **Schema-on-Read** | Схема применяется при чтении (Data Lake) |
| **Data Mesh** | Децентрализованная архитектура данных по доменам |
