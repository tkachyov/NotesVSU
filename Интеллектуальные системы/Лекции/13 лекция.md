# 13 Графы знаний (Knowledge Graphs): определение, стандарты и применение

## 13.1 Блок 1 — KG: определение и место в AI-ландшафте

### 13.1.1 Формальное определение

**G = (E, R, T) — граф знаний**, где:
- **E** — множество сущностей (entities)
- **R** — типы отношений (relation types)
- **T ⊆ E × R × E** — триплеты (факты)

### 13.1.2 Происхождение термина

| Год | Событие |
|-----|---------|
| **2012** | Термин «Knowledge Graph» введён компанией Google |
| **Назначение** | Система, связывающая факты о реальных объектах: людях, местах, организациях, произведениях |
| **Результат** | Knowledge Panels в поисковой выдаче |

### 13.1.3 KG vs онтология

> **KG = онтология + данные в масштабе**

| Компонент | Онтология (TBox) | KG (ABox) |
|-----------|------------------|-----------|
| **Содержание** | Аксиомы о классах и свойствах (схема домена) | Утверждения об индивидах (данные) |
| **Пример** | `Film ⊑ MediaProduct`, `∃hasDirector.Person` | `:Terminator rdf:type :ActionFilm` |
| **Частота изменений** | Меняется редко («теория») | Меняется постоянно («факты») |
| **Масштаб** | Небольшой | Миллиарды триплетов |

### 13.1.4 KG в промышленном масштабе

| Характеристика | Описание |
|----------------|----------|
| **Масштабируемость** | Миллиарды триплетов в распределённых хранилищах |
| **Стандарты** | RDF/OWL/SPARQL — машинная и человеческая читаемость |
| **Связанность** | Ссылки между KG через общие URI — Linked Data |
| **Интеграция** | API, SPARQL-эндпоинты, подключение к LLM/RAG-системам |

### 13.1.5 Пример KG для кинопрокатного домена

**Сущности**: фильмы, режиссёры, жанры, кинотеатры
**Отношения**: `has-director`, `belongs-to-genre`, `screened-in`
**Факты**:
- (Терминатор, has-director, Кэмерон)
- (Терминатор, belongs-to-genre, Боевик)
- (Терминатор, released-in, 1984)

> *Wikidata представляет фильм «Терминатор» как Q127367 с десятками свойств*

## 13.2 Semantic Web Layer Cake: стек стандартов W3C

Слои (снизу вверх):

| Уровень | Стандарт | Назначение |
|---------|----------|------------|
| **Синтаксис** | URI / IRI | Идентификация ресурсов |
| **Обмен** | XML / Turtle / JSON-LD | Сериализация данных |
| **Модель данных** | RDF | Триплеты (субъект-предикат-объект) |
| **Схема** | RDFS | Классы, свойства, иерархии |
| **Семантика** | OWL | Дескрипционные логики, reasoner |
| **Запросы** | SPARQL | SQL-подобный язык для RDF |
| **Безопасность** | RDF* / SHACL | Валидация правил |

## 13.3 Блок 2 — RDF: модель данных

### 13.3.1 URI и пространства имён

**Проблема**: как избежать коллизий имён в распределённой системе?
**Решение**: URI (Uniform Resource Identifier) — каждый ресурс имеет глобально уникальный адрес.

**Основные префиксы**:

| Префикс | URI | Использование |
|---------|-----|---------------|
| `rdf:` | `http://www.w3.org/1999/02/22-rdf-syntax-ns#` | Базовые типы: `type`, `subject`, `Statement` |
| `rdfs:` | `http://www.w3.org/2000/01/rdf-schema#` | Схема: `Class`, `subClassOf`, `domain`, `range` |
| `owl:` | `http://www.w3.org/2002/07/owl#` | Онтология: `equivalentClass`, `ObjectProperty` |
| `xsd:` | `http://www.w3.org/2001/XMLSchema#` | Типы данных: `string`, `integer`, `gYear`, `float` |
| `schema:` | `https://schema.org/` | Schema.org для общих сущностей (Movie, director) |

### 13.3.2 Turtle: наиболее читаемый синтаксис RDF

```turtle
@prefix film: <http://example.org/film/> .
@prefix schema: <https://schema.org/> .
@prefix xsd: <http://www.w3.org/2001/XMLSchema#> .

film:Terminator a schema:Movie ;
    schema:name "The Terminator" ;
    schema:datePublished "1984"^^xsd:gYear ;
    schema:director film:JamesCameron ;
    schema:actor film:Schwarzenegger , film:LindaHamilton ;
    schema:genre film:SciFi , film:Action ;
    schema:aggregateRating "8.0"^^xsd:float .
```

**Синтаксические элементы**:
| Символ | Значение |
|--------|----------|
| `a` | Сокращение для `rdf:type` |
| `;` | Продолжение того же субъекта с новым предикатом |
| `,` | Тот же субъект + тот же предикат, другой объект |
| `.` | Завершение всех утверждений об этом субъекте |
| `^^` | Явный тип данных литерала (из XSD) |

## 13.4 Блок 3 — SPARQL: язык запросов

### 13.4.1 Четыре типа запросов

| Тип | Назначение | Аналог |
|-----|------------|--------|
| **SELECT** | Возвращает таблицу значений переменных | SQL SELECT |
| **CONSTRUCT** | Создаёт новый RDF-граф из паттернов | Трансформация данных |
| **ASK** | Булев ответ (существует ли хотя бы один результат) | EXISTS |
| **DESCRIBE** | Возвращает все известные факты об указанном ресурсе | — |

### 13.4.2 Basic Graph Pattern (BGP)

**BGP** — набор триплет-паттернов с переменными (`?var`). Аналог `WHERE` + `JOIN` в SQL.

**Пример запроса** (фильмы с режиссёрами):
```sparql
PREFIX schema: <https://schema.org/>
PREFIX film: <http://example.org/film/>

SELECT ?title ?directorName WHERE {
    ?film a schema:Movie ;
          schema:name ?title ;
          schema:director ?director .
    ?director schema:name ?directorName .
}
LIMIT 10
```

**Как работает BGP**:
1. Движок ищет все тройки, соответствующие первому паттерну (`?film a schema:Movie`)
2. Для каждого `?film` ищет паттерны с его URI в позиции субъекта
3. Переменная `?director` связывает два паттерна — это **неявный JOIN**
4. Возвращает только строки, где все паттерны удовлетворены одновременно

### 13.4.3 Пример для домена «Кинопрокат»

**Запрос**: все боевики с рейтингом выше 7.5
```sparql
SELECT ?title ?rating WHERE {
    ?film a film:ActionFilm ;
          schema:name ?title ;
          schema:aggregateRating ?rating .
    FILTER(?rating > 7.5)
}
ORDER BY DESC(?rating)
```

## 13.5 Блок 4 — Reasoning на KG

### 13.5.1 Forward Chaining (материализация)

Ризонер заранее добавляет все логически выводимые факты в граф.

**Процесс**:

| Шаг | Действие |
|-----|----------|
| **1** | Исходный граф + аксиомы TBox |
| **2** | Применение правил вывода (например, `subClassOf`: если `X a C` и `C rdfs:subClassOf D`, то `X a D`) |
| **3** | Материализованные факты добавляются в граф |
| **4** | Обычные SPARQL-запросы находят выведенные факты как обычные данные |

**Плюсы и минусы**:

| Плюсы | Минусы |
|-------|--------|
| Запросы быстрые | Граф вырастает в 5–10 раз |
| | При обновлении онтологии нужен пересчёт всего вывода |

### 13.5.2 Backward Chaining (query rewriting)

Запрос переписывается с учётом аксиом во время выполнения — граф не меняется.

**Пример** (исходный запрос):
```sparql
SELECT ?x WHERE { ?x rdf:type :Film }
```

**Переписанный** (с учётом `subClassOf`):
```sparql
SELECT ?x WHERE {
    { ?x rdf:type :Film }
    UNION { ?x rdf:type :ActionFilm }
    UNION { ?x rdf:type :Documentary }
}
```

**Сравнение подходов**:

| Критерий | Forward (материализация) | Backward (query rewriting) |
|----------|--------------------------|----------------------------|
| **Скорость запросов** | ✅ Быстрая | ❌ Медленнее |
| **Размер графа** | ❌ Разрастается | ✅ Не меняется |
| **Актуальность** | ❌ Требует пересчёта | ✅ Всегда актуальна |
| **Сложность при обновлении** | ❌ Высокая | ✅ Низкая |

### 13.5.3 Промышленная практика: гибридный подход

- **Материализация** для «горячих» паттернов (`rdfs:subClassOf`, `owl:equivalentClass`)
- **Query rewriting** для редких аксиом
- **Примеры**: Stardog (configurable), GraphDB (forward с incremental update)

## 13.6 Блок 6 — Хранилища KG

### 13.6.1 Архитектурные стратегии

| Тип | Описание | Примеры |
|-----|----------|---------|
| **Нативные RDF-хранилища** | Спроектированы для триплетов с нуля; индексные структуры под SPARQL (6 перестановок SPO-индексов) | Stardog, GraphDB, Virtuoso, Jena TDB |
| **Реляционные СУБД + RDF-слой** | Триплеты мапятся на реляционные таблицы; подходы: Triple Table, Vertical Partitioning, Property Tables | Ontop, D2RQ |

### 13.6.2 Ключевой вопрос выбора

| Сценарий | Рекомендация |
|----------|--------------|
| Много SPARQL-запросов с произвольными паттернами | **Нативный триплетор** |
| Данные уже в реляционной БД, SPARQL как дополнение | **Реляционный адаптер** |
| Граф-обходы и алгоритмы (PageRank, кратчайший путь) | **LPG** (Neo4j, Amazon Neptune) |
| Облачная среда с managed-сервисом | Amazon Neptune, Azure Cosmos DB Gremlin |

## 13.7 Блок 8 — KG в современном AI-стеке

### 13.7.1 KG + RAG: структурированный контекст для LLM

**Схема**:
```
Пользовательский вопрос → Entity Linking → SPARQL → Факты → LLM-контекст → Ответ
```

**Преимущество KG-RAG перед pure vector RAG**:

| Преимущество | Описание |
|--------------|----------|
| **Верифицированные факты** | Нет галлюцинаций на структурированных данных (имена, даты, отношения) |
| **Прозрачность** | SPARQL-запрос — явная цепочка рассуждений (можно проверить и залогировать) |
| **Актуальность** | KG обновляется независимо от модели — свежие данные без дообучения LLM |

**Пример для кинопроката**: чат-бот кинотеатра на вопрос *«Когда следующий сеанс Аватара в зале 5?»* выполняет entity linking (Avatar, Hall5) → SPARQL-запрос к расписанию KG → точный ответ без галлюцинаций.

### 13.7.2 LLM → KG: автоматическое наполнение графа

**Процесс**:

| Шаг | Действие | Инструменты |
|-----|----------|-------------|
| **1** | Текст → сущности (NER) | REBEL, spaCy, OpenIE |
| **2** | Сущности → отношения (RE) | REBEL, LLM-prompting |
| **3** | Разрешение сущностей (entity disambiguation) | Wikifier, DBpedia Spotlight |
| **4** | Добавление в KG (SHACL-валидация) | Stardog, GraphDB |

**Пример**:
- Текст: *«Джеймс Кэмерон снял „Аватар“ в 2009 году»*
- NER: `:JamesCameron (Person)`, `:Avatar (Film)`, `2009 (Year)`
- RE: `(:JamesCameron, directed, :Avatar)`, `(:Avatar, releaseYear, 2009)`
- Disambiguation: «Кэмерон» → Q42574 (WikiData)

**Основная проблема**: точность для редких отношений низкая (~60–70%). Галлюцинация LLM при RE — критичный риск.

### 13.7.3 GraphRAG (Microsoft, 2024): KG + Vector DB

**Идея**: не просто векторы, а граф сущностей + community summaries для глобальных запросов.

**Архитектура**:

| Этап | Действие |
|------|----------|
| **1. Индексация** | Текстовый корпус → LLM извлекает сущности + отношения → строится KG документа → алгоритм Leiden выделяет «сообщества» связанных сущностей |
| **2. Community summaries** | LLM генерирует обобщение для каждого сообщества («Эти 20 сущностей связаны темой X») → иерархические summaries |
| **3. Глобальные запросы** | Вопрос → релевантные community summaries → LLM синтезирует ответ из нескольких summaries → лучше для вопросов типа «каковы главные темы?» |
| **4. Локальные запросы** | Традиционный vector RAG по конкретным сущностям + KG для навигации |

## 13.9 Ключевые термины лекции

| Термин | Определение |
|--------|-------------|
| **Knowledge Graph (KG)** | G = (E, R, T) — граф знаний с сущностями, отношениями и триплетами |
| **TBox** | Terminological Box — аксиомы о классах и свойствах (онтология) |
| **ABox** | Assertional Box — утверждения об индивидах (данные) |
| **RDF** | Resource Description Framework — модель данных (триплеты) |
| **RDFS** | RDF Schema — схема для RDF (классы, свойства, иерархии) |
| **OWL** | Web Ontology Language — язык онтологий на основе дескрипционных логик |
| **SPARQL** | Язык запросов к RDF-графам (SELECT, CONSTRUCT, ASK, DESCRIBE) |
| **BGP** | Basic Graph Pattern — паттерн сопоставления триплетов с переменными |
| **Forward Chaining** | Материализация — вывод добавлением фактов в граф |
| **Backward Chaining** | Query rewriting — запрос переписывается с учётом аксиом |
| **GraphRAG** | Microsoft 2024 — KG + векторная БД + community summaries |
| **SHACL** | Shapes Constraint Language — валидация RDF-графов |
